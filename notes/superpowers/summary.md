# Superpowers 项目总结

来源：[obra/superpowers](https://github.com/obra/superpowers)

## 这个项目是什么

`superpowers` 不是一个普通的业务应用，也不是单一 SDK 或工具库，而是一套给 AI 编码代理使用的 **软件开发工作流增强系统**。

它的核心思路是：不要让代理一收到需求就直接写代码，而是先通过一组可组合、可自动触发的 `skills`，把需求澄清、设计、计划、测试、代码审查、分支收尾这些步骤串成一条更稳定的研发流水线。

README 里对它的定位说得很直接：这是一个建立在多种 composable skills 之上的完整软件开发工作流，外加一层初始指令，用来确保代理真的会去用这些技能，而不是把它们当成“可选提示词”。

如果用一句话概括：

**Superpowers 想做的，不是给代理补一个新功能，而是给代理补一套更像成熟工程团队的工作方式。**

## 核心定位

这个项目主要承担四种角色：

- **代理开发流程框架**：定义从需求讨论到开发完成的完整链路
- **技能库**：把不同研发环节拆成可独立触发的 `SKILL.md`
- **插件/扩展分发仓库**：支持 Claude Code、Cursor、Codex、OpenCode、Gemini CLI 等环境安装
- **方法论实现仓库**：把 TDD、系统化调试、计划驱动执行、子代理协作等工程实践写成可执行流程

所以它和很多“提示词仓库”的差别在于：

- 它强调 **workflow**，不是只强调 prompt
- 它强调 **强约束执行顺序**，不是只提供建议
- 它强调 **持续验证和 review gate**，不是一次性生成结果
- 它强调 **子代理分工**，不是让一个大代理包打天下

## 项目的基本工作流

这个仓库想让代理按下面的顺序工作：

1. **brainstorming**
   在真正写代码前先澄清需求、探索方案、形成设计，并把设计写成文档
2. **using-git-worktrees**
   在隔离工作区和分支里开展实现，减少污染主分支的风险
3. **writing-plans**
   把设计拆成很细的、可执行的步骤，每一步都要写明文件、测试、命令和预期结果
4. **subagent-driven-development** 或 **executing-plans**
   按计划执行，其中优先推荐子代理驱动模式
5. **test-driven-development**
   所有功能和修复都要求先写失败测试，再写最小实现
6. **requesting-code-review / receiving-code-review**
   在任务间和完成前做审查闭环
7. **finishing-a-development-branch**
   最后验证、整理、合并或发 PR

也就是说，`superpowers` 本质上是在把“设计驱动 + 计划驱动 + TDD + review gate + branch hygiene”组合成一个完整体系。

## 仓库结构

从仓库内容看，核心部分主要有这些：

- `skills/`
  这是仓库最重要的目录。每个子目录就是一个技能，核心文件是 `SKILL.md`。
- `docs/`
  放不同平台的接入说明、设计文档和测试说明。
- `tests/`
  放技能级测试、集成测试和辅助脚本，尤其强调对复杂工作流进行真实验证。
- `hooks/`
  放会话启动和工作流注入相关的 hook。
- 平台接入文件
  例如 `gemini-extension.json`，说明这个仓库不只是文档集合，而是面向多个代理运行环境的实际插件包。

这说明它不是单纯“展示 skill 怎么写”，而是已经把 skill 的发现、装载、注入、测试和跨平台分发都考虑进去了。

## 最重要的技能分层

如果把 `superpowers` 里的能力按职责分层，大概可以分成下面几类：

### 1. 流程入口与调度层

- `using-superpowers`
- `brainstorming`
- `writing-plans`
- `executing-plans`
- `subagent-driven-development`

这一层决定代理 **什么时候应该先思考、什么时候该写计划、什么时候该派子代理执行**。它们管的是“流程怎么走”。

### 2. 实施约束层

- `test-driven-development`
- `systematic-debugging`
- `verification-before-completion`

这一层决定代理 **写代码、修 bug、收尾验证时要遵守什么纪律**。它们管的是“做事时要按什么标准做”。

### 3. 协作与质量控制层

- `requesting-code-review`
- `receiving-code-review`
- `dispatching-parallel-agents`
- `finishing-a-development-branch`
- `using-git-worktrees`

这一层负责 **把质量关口、并行协作和分支管理纳入流程**。

### 4. 元能力层

- `writing-skills`
- `using-superpowers`

这一层服务于“如何继续扩展这套体系本身”。

这说明 `superpowers` 不是“很多零散技能堆在一起”，而是有明显的工作流分层设计。

## 这个项目最有特点的地方

读下来，这个仓库最值得注意的不是某个具体 skill，而是它体现出的几种工程取向。

### 1. 它想解决的是“代理太快开始写代码”

很多 AI 编码助手的问题并不是不会写代码，而是：

- 没搞清需求就开始写
- 没确定设计就开始改
- 没计划就跨很多文件乱动
- 没测试就自称完成
- 没 review 就继续往下做

`superpowers` 的所有流程几乎都在解决这个问题。它通过硬性顺序把“想清楚再动手”写进代理行为里。

### 2. 它强调强约束，而不是“建议”

这个项目里很多技能都用了非常强的约束语言，例如：

- 先 brainstorming，没过设计评审就不能实现
- 没有 failing test 就不能写生产代码
- review 顺序必须先 spec compliance，再 code quality
- 没过 review 不能进入下一步

这让它更像 **流程控制系统**，而不是“提示增强包”。

### 3. 它把“上下文管理”看得非常重要

在 `subagent-driven-development` 里，核心思想之一就是：

- 每个任务使用新的子代理
- 子代理不继承主会话的历史上下文
- 主代理只给它完成任务所需的精确上下文

这背后的理念很清楚：**上下文污染会降低代理执行质量**。所以它不是一味追求“更大上下文”，而是追求“更干净、可控的上下文”。

### 4. 它不仅关注质量，也关注可扩展和可测试

仓库里不只是有 skill，还有针对 skill 的测试、集成测试和 token 分析工具。这点很关键，因为它说明作者把这套工作流当成一个要维护、要迭代、要验证的产品，而不是一次性的经验总结。

## 这个项目和 Anthropic `skills` 仓库的关系

如果把你前面看的 `anthropics/skills` 和这个项目对照起来，可以这样理解：

- `anthropics/skills` 更像 **官方示例库 / 参考实现库**
- `superpowers` 更像 **围绕软件开发任务定制出来的一整套代理研发流程**

前者更强调“Skill 可以怎么写、能做什么”，后者更强调“多个 Skill 怎么组合成严密的工程工作流”。

所以 `superpowers` 并不是在重复 Anthropic 的示例仓库，而是在沿着 Skill 体系继续往前走，专门解决“AI 代理如何更像靠谱工程师地做开发”这个问题。

## 这个项目和规格驱动开发（SDD）的对比

如果拿它和规格驱动开发（Spec-Driven Development, SDD）对照，二者确实有明显相似性，但关注点并不完全一样。

相似点主要在于：

- 都反对“一上来就写代码”
- 都强调先把需求、约束、验收标准说清楚
- 都希望实现过程能围绕明确规格，而不是围绕临场猜测
- 都把“规格是否被满足”当成质量判断的重要依据

但差别也很明显：

- **SDD 的中心是 spec**：先产出规格，再根据规格设计、实现和验证
- **`superpowers` 的中心是 workflow orchestration**：不只要求有规格，还把 brainstorming、计划拆解、TDD、子代理协作、review gate、分支收尾一起编排进流程
- **SDD 更像一种开发原则**：强调“以规格作为开发和验收的主线”
- **`superpowers` 更像一种代理执行系统**：强调“让 AI 代理按什么顺序、用什么技能、经过哪些关卡来完成任务”

换句话说，SDD 解决的是“开发应该围绕什么展开”，而 `superpowers` 解决的是“代理应该按什么工作流把这件事做稳”。

所以可以把两者理解成：

- **SDD 提供目标对象**：先把 spec 定义清楚
- **`superpowers` 提供执行机制**：再把 spec、计划、测试和 review 串成可操作流程

如果硬要归类，`superpowers` 并不是纯粹的 SDD 框架，但它明显吸收了 SDD 的思想，并把这种思想进一步工程化成了适合 AI 编码代理执行的工作流系统。

## 这个项目的实际价值

如果你是开发者，这个项目的价值主要在这些方面：

- **给 AI 编码流程加护栏**：避免代理无计划、大范围、无验证地乱改代码
- **提升复杂任务稳定性**：把大任务拆成设计、计划、实现、审查多个阶段
- **提升可复用性**：把研发经验写成可重复触发的 skill
- **更适合长任务**：尤其是需要连续工作几十分钟甚至几小时的任务
- **更适合多人/多代理协作思路**：把单代理执行改造成主代理 + 子代理的组织模式

它尤其适合：

- 想让 AI 在真实代码库里做中大型任务的人
- 想把团队工程规范“写进代理行为”的人
- 想研究 agent workflow 而不是只研究 prompt 的人

## 局限和代价

这套体系也不是没有成本。

### 1. 流程更重

相比“直接让 AI 改文件”，`superpowers` 会明显更慢，因为它会先做设计、计划、测试、review，再推进实现。

### 2. token 和调用成本更高

特别是子代理驱动模式里，一个任务可能会触发：

- 1 个实现子代理
- 1 个 spec reviewer
- 1 个 code quality reviewer
- 必要时还会多轮返工和重审

质量更稳，但成本一定更高。

### 3. 更依赖宿主平台能力

如果宿主环境不支持 subagent，或者工具能力不够，这套流程的效果会打折。这也是为什么仓库里明确区分了 `subagent-driven-development` 和 `executing-plans` 两条执行路线。

### 4. 对任务纪律要求高

这套方法不适合“随便试试”。如果使用者和代理都不愿意遵守它要求的顺序，效果会明显下降。

## 一句话总结

`superpowers` 是一个面向 AI 编码代理的工程化工作流仓库。它的核心价值不在于增加某个单点能力，而在于把设计、计划、TDD、子代理协作、代码审查和分支收尾整合成一套高约束、高验证、可跨平台复用的软件开发方法。
