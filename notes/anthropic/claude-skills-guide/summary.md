# Building Skills for Claude 指南总结

来源：<https://resources.anthropic.com/hubfs/The-Complete-Guide-to-Building-Skill-for-Claude.pdf>

## 这份指南主要讲什么

这是一份关于如何为 Claude 构建 Skill 的完整实践指南。核心目标不是单纯介绍文件格式，而是教你把“可重复的工作流经验”打包成一个可复用的能力模块，让模型在合适的时候自动加载，并按稳定流程执行任务。

文中强调，Skill 本质上是“知识层”，而 MCP 更像是“工具层”：

- MCP 决定模型“能做什么”
- Skill 决定模型“应该怎么做”

两者结合后，Claude 才能稳定完成多步骤、强约束、有领域经验要求的任务。

## Skill 的基本结构

一个 Skill 通常是一个文件夹，最少只需要一个 `SKILL.md` 文件，其他内容按需补充：

- `SKILL.md`：必需，包含 YAML frontmatter 和主说明
- `scripts/`：可选，放校验脚本、处理脚本等
- `references/`：可选，放详细文档、范例、接口说明
- `assets/`：可选，放模板、图标、字体等资源

几个硬性要求：

- 文件名必须严格叫 `SKILL.md`
- Skill 文件夹名要用 kebab-case
- 文件夹里不要放 `README.md`
- frontmatter 里必须至少有 `name` 和 `description`

## 最关键的是 frontmatter

指南反复强调：Skill 能不能被正确触发，最关键是 `description`。

一个有效的 `description` 必须同时说明两件事：

- 这个 Skill 是做什么的
- 用户在什么场景下应该使用它

推荐写法是：

`[它做什么] + [什么时候使用] + [典型触发词/任务]`

好的描述应当：

- 具体，不抽象
- 包含用户真实会说的话
- 尽量带上任务类型、对象、文件类型或业务上下文
- 避免只写技术实现，不写使用场景

坏的描述通常过于模糊，比如只是写“帮助处理项目”。

## Skill 设计原则

指南给了三个核心原则：

### 1. Progressive Disclosure

不要把所有说明都塞进主文件。Skill 应该分层：

- 第一层：frontmatter，提供最小触发信息
- 第二层：`SKILL.md` 正文，提供主要流程说明
- 第三层：`references/` 等补充材料，按需再读

这样能减少上下文占用，提高触发准确率和执行效率。

### 2. Composability

Skill 不应该假设自己是唯一生效的能力。Claude 可能同时加载多个 Skill，所以指令设计要可组合、不要互相冲突。

### 3. Portability

Skill 最好能跨 Claude.ai、Claude Code 和 API 使用，尽量不要依赖只在某个环境才成立的假设。

## 开始写 Skill 前，先定义用例

文中建议，先不要急着写说明，而是先确定 2 到 3 个具体用例。每个用例最好明确：

- 用户想完成什么
- 触发语句可能是什么
- 需要哪些步骤
- 依赖哪些工具或 MCP
- 最终成功结果是什么

如果连用例都说不清，Skill 大概率也写不好。

## 常见的三类 Skill

### 1. 文档与资产生成类

适合生成文档、页面、设计稿、代码、报告等内容，重点在输出质量和风格一致性。

### 2. 工作流自动化类

适合处理一套固定步骤的流程，比如创建项目、整理数据、生成报告、执行检查等，重点在流程清晰和步骤稳定。

### 3. MCP 增强类

适合给已有 MCP 工具补上“最佳实践”。MCP 只提供工具访问，Skill 则补充任务编排、调用顺序、领域规则和错误处理。

## 如何写出有效的主说明

`SKILL.md` 正文建议聚焦“怎么做”，而不是大段背景介绍。推荐结构大致如下：

- Skill 名称
- Instructions
- 分步骤流程
- 示例场景
- 常见问题与排错

好的说明应该满足：

- 可执行，而不是泛泛建议
- 明确每一步做什么
- 关键步骤写清输入、输出和校验方式
- 对常见错误给出原因和解决方案
- 把过长细节移到 `references/`

文中尤其强调：模糊语句很容易让模型执行不稳定。相比“请先做好验证”，更好的写法是列出具体校验项。

## 五种常见设计模式

指南总结了几类高频模式，适合直接套用：

### 1. 顺序工作流编排

适合必须按步骤执行的流程。每一步写清依赖关系、前置条件和失败后的处理。

### 2. 多 MCP 协同

适合跨多个服务联动，比如从设计工具导出资源、上传存储、创建任务、再通知团队。

### 3. 迭代优化

适合需要“生成 -> 检查 -> 修复 -> 再验证”的任务，例如报告生成、代码修复、质量审查等。

### 4. 上下文感知的工具选择

适合同一目标下，根据文件类型、大小、协作场景等选择不同工具。

### 5. 领域知识增强

适合金融合规、客服流程、内部规范等场景，把业务规则直接嵌进 Skill 的决策过程。

## 测试 Skill 的三条主线

指南建议至少从三个维度测试：

### 1. 触发测试

检查：

- 明显相关请求是否能触发
- 换一种说法是否也能触发
- 不相关请求是否不会误触发

### 2. 功能测试

检查：

- 输出是否正确
- 工具调用是否成功
- 异常处理是否可用
- 边界情况是否覆盖

### 3. 性能对比

对比有无 Skill 时：

- 需要多少轮对话
- 失败调用是否减少
- 总 token 是否下降
- 用户是否还需要频繁补充说明

文中很强调一个实践：先围绕一个困难但典型的任务反复迭代，做通以后再扩展到更多用例。

## 最常见的问题与对应思路

### 1. 上传失败

通常是以下原因：

- 没有正确命名为 `SKILL.md`
- YAML frontmatter 格式错误
- Skill 名称不符合 kebab-case

### 2. Skill 不触发

最常见原因不是模型能力不够，而是 `description` 太泛、太抽象、没有触发词。

### 3. Skill 触发过多

说明范围写得太宽，需要增加更具体的限制条件，甚至加入“不要用于什么场景”的负向描述。

### 4. MCP 调用失败

要区分是 Skill 问题还是 MCP 本身没连好。先单独验证 MCP 能否工作，再检查 Skill 里工具名和调用流程是否正确。

### 5. 指令不被稳定遵守

常见原因：

- 说明太长
- 关键要求埋得太深
- 语言不够明确
- 把本该脚本化的校验只写成自然语言要求

指南建议：关键校验能用脚本就尽量用脚本，因为脚本比文字更确定。

## 关于发布与分发

如果要公开共享 Skill，文中建议：

- 用 GitHub 仓库存放
- 在仓库根目录写给人看的 `README.md`
- Skill 文件夹内部仍然不要放 `README.md`
- 在 MCP 文档里说明“为什么 Skill + MCP 组合更有价值”
- 给出简洁安装说明和示例

同时，文中提到 Skills 已经在逐步形成开放标准，未来不仅限于 Claude.ai 手动上传，也可以通过 API 和组织级管理方式分发。

## 这份指南最值得记住的结论

可以把它概括成下面几点：

1. Skill 不是提示词备忘录，而是面向任务的可复用工作流封装。
2. 决定 Skill 是否好用的关键，不只是说明写得多，而是触发描述是否准确、步骤是否可执行。
3. Skill 最适合处理“有固定流程、需要领域经验、希望结果一致”的任务。
4. Skill 和 MCP 是互补关系：MCP 给能力，Skill 给方法。
5. 真正高质量的 Skill 一定是测试和迭代出来的，而不是一次写完。

## 如果要基于这份指南落地实践

一个可执行的最小流程可以是：

1. 先挑 2 到 3 个高频、重复、步骤明确的任务
2. 为每个任务写出触发语句、步骤、依赖工具和成功标准
3. 先写一个最小版 `SKILL.md`
4. 重点打磨 `description`
5. 做触发测试、功能测试、误触发测试
6. 把长说明拆到 `references/`，把关键校验放到 `scripts/`
7. 根据真实失败案例继续迭代

## 适合这份指南的读者

这份文档尤其适合以下人群：

- 想把常用提示流程沉淀成固定能力的人
- 已经有 MCP，但希望使用体验更稳定的人
- 想把团队工作流标准化的人
- 想做内部自动化助手或可复用 AI 能力模块的人

## 一句话总结

这份指南的核心思想是：把“会用工具”升级成“会按最佳流程完成任务”，而 Skill 就是把这种流程经验系统化、可复用、可测试地交给 Claude 的方式。
