# 阅读开源项目

## 快速开始

当用户要求阅读一个开源项目时：

1. 先确认目标仓库、目标子系统，以及希望输出的深度。
2. 优先阅读高信号文件：`README*`、根目录配置、依赖清单、文档与顶层目录。
3. 在深入实现前，先建立仓库结构地图。
4. 选择一条具有代表性的端到端主流程进行追踪。
5. 基于文件和符号给出结论，并明确列出未确认的问题。

## 工作流

### 1. 明确范围

- 如果用户没有明确范围，默认采用“仓库总览 + 一条核心流程”。
- 如果仓库较大，分阶段阅读：先总览，再深入。
- 阅读目标要和用户意图对齐，例如架构、启动流程、训练循环、Agent 主循环、插件系统、API 链路或模块设计。

### 2. 建立项目背景

- 识别技术栈、包管理器、运行时以及启动方式。
- 判断项目主要属于库、应用、框架、CLI、Agent 系统、训练系统，还是基础设施组件。
- 尽量复用仓库本身的术语，不要自造概念标签。

### 3. 先建立结构地图

- 在阅读实现细节前，先检查顶层目录和关键配置文件。
- 优先寻找可能的入口点，例如 `main`、`cli`、`server`、`app`、`index`、`routes`、`pipeline`、`trainer`、`worker`、`agent`。
- 至少记录这些信息：入口点、主要模块、关键抽象、状态边界、外部集成。

### 4. 追踪一条核心流程

- 根据用户目标选择一条代表性流程，例如请求处理、Agent 执行循环、检索链路、训练步骤、任务调度或插件注册。
- 跨模块追踪这条流程，并记录：触发点、编排者、核心业务逻辑、IO 边界、错误处理或重试机制、扩展点。

### 5. 提炼设计决策

- 区分“实现细节”和“架构决策”。
- 主动识别 registry、adapter、event bus、DAG、queue、state machine、worker model、tool abstraction 等模式。
- 说明这些模式为什么会存在，以及它们带来的权衡。

### 6. 输出结构化笔记

- 优先做高质量总结，不要机械罗列每个文件。
- 做判断时引用具体文件和符号。
- 如果仓库太大，无法完整阅读，要明确说明未覆盖部分。
- 默认使用 [template.md](template.md) 中的模板。

## 阅读启发式

- 先看对外暴露的表层，再逐步深入内部实现。
- 文档与代码冲突时，以代码为准。
- 跟着数据流和控制流走，不要只看目录名。
- 聚焦能解释行为的最小文件集合。
- 仓库很大时，先找一条“黄金路径”，不要试图一次读完全部内容。

## 注意事项

- 不要只读完 `README` 就声称已经完全理解项目。
- 除非用户明确要求穷举，否则不要逐文件摘要。
- 避免没有证据的空泛表述，例如“采用模块化设计”。
- 如果关键行为来自代码生成或间接配置，要明确说清楚。
- 如果仓库包含多个应用或多个包，要区分共享基础设施与应用特定逻辑。

## 输出要求

除非用户要求其他格式，否则默认使用 [template.md](template.md)。

## 典型触发语句

- “帮我读一下这个开源项目”
- “总结这个仓库的架构”
- “解释这个 Agent 框架主循环怎么跑”
- “梳理这个 RAG 项目的检索链路”
- “告诉我应该按什么顺序读这个仓库”
